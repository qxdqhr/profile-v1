# 闲鱼自动化可行性分析

## 1. 需求拆解

| 操作 | 自动化难度 | 推荐方案 | 备注 |
|------|------------|----------|------|
| 登录/维持会话 | ⭐⭐ | Cookie 导入 / 扫码 | `_m_h5_tk` 需定期刷新 |
| 图片上传 | ⭐⭐ | stream-upload API | 依赖登录态 |
| 类目识别 | ⭐⭐ | kgraph.property.recommend | 可接 AI 增强 |
| 商品发布 | ⭐⭐⭐ | mtop.idle.pc.idleitem.publish | 核心链路 |
| 商品编辑 | ⭐⭐⭐ | 待抓包确认 edit 接口 | |
| 商品下架/删除 | ⭐⭐ | CLI 已有实现 | |
| IM 自动回复 | ⭐⭐⭐⭐ | WebSocket + 消息协议 | msgpack 解包 |
| 订单管理 | ⭐⭐ | 官方 API（ISV）或 Mtop | 个人卖家走 Mtop |
| 批量多账号 | ⭐⭐⭐⭐ | Cookie 池 + 代理 IP | 风控压力大 |

---

## 2. 三条技术路线

### 路线 A：官方 TOP API（ISV）

```
企业入驻 → 闲鱼运营准入 → AppKey + OAuth → TOP 网关调用
```

**优点**

- 合规、稳定、有官方文档
- 适合已验货、同城、租赁等垂直场景
- 订单/退款/发货链路完整

**缺点**

- 个人开发者无法申请
- 不包含「普通 C 端闲置发品」场景
- 聚石塔部署、企业资质要求

**可行性**：❌ 不适合个人闲置自动化；✅ 适合签约 ISV 业务

---

### 路线 B：Web Mtop 协议（推荐 PoC）

```
浏览器登录 → 导出 Cookie → Python/Node 客户端 → H5 签名 → Mtop 网关
```

**优点**

- 签名算法简单（MD5）
- 开源参考多（XianYuApis、goofish-cli、myfish）
- 覆盖 PC 发品全链路
- 无需 Android 设备/Frida

**缺点**

- 违反平台 ToS 风险
- Cookie 过期需维护
- 419 滑块/x5sec 需人工或打码
- 接口无文档，随版本可能变化

**可行性**：✅ **个人卖家批量发品最现实的路径**

**最小 PoC 步骤：**

1. 浏览器登录 goofish.com，导出 Cookie
2. 实现 `generate_sign()` + `_m_h5_tk` 刷新
3. 调用 `upload.api` 上传 1 张图
4. 调用 `kgraph.property.recommend` 获取类目
5. 调用 `local.poi.get` 获取地址
6. 调用 `idleitem.publish` 发布测试商品
7. 验证发布后商品可在 App 中看到

**预估工作量**：1-2 人周（含 Cookie 管理、错误处理）

---

### 路线 C：App Mtop + 签名服务

```
Android 设备 + Frida/Unidbg → 签名 RPC 服务 → Python 业务层
```

**优点**

- 功能最全（App 独有接口）
- IM/WebSocket 协议支持更好（myfish）

**缺点**

- 签名逆向维护成本高
- 每个 App 大版本可能 break
- 需真机或模拟器常驻
- Native 模块编译/保护

**可行性**：⚠️ 适合 IM Bot、深度运营；发品可优先 Web

---

### 路线 D：浏览器自动化（Playwright）

```
Playwright 持久 Context → 模拟点击发布页 → 填表上传
```

**优点**

- 不依赖签名逆向
- 行为最接近真实用户
- 滑块可人工介入

**缺点**

- 速度慢、资源占用高
- 页面 DOM 变化需维护选择器
- 大规模批量不现实
- 仍可能被行为风控检测

**可行性**：✅ 适合低频次、半自动；❌ 不适合大规模

参考：xianyu-openclaw-channel 的 Playwright 发品方案

## 3. 方案对比矩阵

| 维度 | 官方 API | Web Mtop | App Mtop | Playwright |
|------|----------|----------|----------|------------|

| 发品 | ⚠️ ISV only | ✅ | ✅ | ✅ |

| 风控风险 | 低 | 中-高 | 中-高 | 中 |
| 批量能力 | 高 | 高 | 高 | 低 |

## 4. 推荐架构（Web Mtop 路线）

```
┌─────────────────────────────────────────────────┐
│                   业务层                         │
│  发品调度 / 模板管理 / 多账号 / 任务队列          │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│              XianyuClient (SDK)                  │
│  ├─ AuthManager      (Cookie/Token 刷新)         │
│  ├─ MediaUploader    (图片上传)                  │
│  ├─ CategoryService  (类目推荐)                  │
│  ├─ ItemPublisher    (发品/编辑/下架)            │
│  ├─ ImClient         (可选: WS 消息)             │
│  └─ RiskGuard        (频率限制/违禁词/熔断)       │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│              Mtop Transport                      │
│  sign(MD5) + HTTP POST + Cookie Jar              │
└─────────────────────┬───────────────────────────┘
                      │
              h5api.m.goofish.com
              stream-upload.goofish.com
```

### 关键模块说明

**AuthManager**

- 存储多账号 Cookie（加密）
- 检测 `_m_h5_tk` 过期并自动刷新
- 419 时标记账号「需人工过滑块」

**RiskGuard**

- 发布间隔：建议 ≥ 30s，批量间 ≥ 5-15s 随机
- 单账号日发布上限：待实测（建议初始 ≤ 20/天）
- 违禁词/外联词扫描（参考 goofish-cli）
- 连续失败熔断

## 5. 功能可行性结论

| 功能 | 可行 | 推荐实现 | 风险等级 |
|------|------|----------|----------|
| 单账号 PC 发品 | ✅ | Web Mtop | 中 |
| 批量发品（多 SKU） | ✅ | 队列 + 限速 | 高 |
| 多账号矩阵 | ⚠️ | Cookie 池 + 代理 | 很高 |
| 自动类目 | ✅ | kgraph + AI | 低 |
| 图片批量上传 | ✅ | upload.api | 低 |
| 自动下架 | ✅ | Mtop/CLI | 中 |
| IM 自动回复 | ⚠️ | myfish/WS | 高 |
| 自动发货 | ⚠️ | 官方需 ISV；Mtop 待调研 | 中 |
| 商品数据爬取 | ✅ | detail API | 中 |

---

## 6. 不建议的做法

1. **高频批量发品**（>50/天/账号）— 极易触发限流/封号
2. **纯协议不带行为随机** — 419 滑块无法绕过
3. **多账号同 IP 无隔离** — 关联封号
4. **对外提供 SaaS 发品服务** — 法律与平台合规风险
5. **硬逆向 App 签名而不做版本锁定** — 维护地狱

---

## 7. 建议的验证计划

### Phase 1：技术验证（3-5 天）

- [ ] Cookie 登录 + token 刷新跑通
- [ ] 单图上传成功
- [ ] 发布 1 个测试商品（0.01 元或下架）
- [ ] 确认 App 端可见

### Phase 2：稳定性（1 周）

- [ ] 连续 10 次发品成功率
- [ ] 419 滑块触发阈值摸底
- [ ] Cookie 有效期与刷新策略
- [ ] 下架/删除链路

### Phase 3：产品化（可选）

- [ ] 多账号 Cookie 管理 UI
- [ ] 模板化发品（标题/描述/价格/邮费）
- [ ] 任务队列与失败重试
- [ ] IM 接入（如需客服 Bot）
